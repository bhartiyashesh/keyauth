<objective>
Build the service worker (WebSocket client + keepalive + message routing) and the popup UI (QR pairing + connected state + Request Code flow). This is the main interactive plan -- it wires the extension to the relay and gives the user the pairing and code-request experience.

Purpose: Connect the extension to the relay server, implement QR-based pairing that iOS can scan, and enable the user to request a code with one click. This plan covers PAIR-01 (QR generation), PAIR-03 (single-use TTL tokens), PAIR-05 (status indicator), and CODE-01 (Request Code button).

Output: A working service worker maintaining a WebSocket to the relay with 20s pings, and a React popup with QR pairing and Request Code states.
</objective>

<threat_model>
- **T1: Pairing token replay** -- QR encodes a single-use roomId+keypair. After iOS joins and pairing completes, the service worker clears `pendingPairing` from session storage and stores only the derived sharedKey. A second scan of the same QR would attempt to join a room already occupied by 2 clients and be rejected by the relay (room_full).
- **T2: Private key exposure window** -- The X25519 privateKey is held in chrome.storage.session during the pairing window (max 5 minutes). After pairing completes, it is deleted. Session storage is in-memory only and not persisted to disk.
- **T3: WebSocket origin spoofing** -- The relay validates roomId on connection. The relay URL is hardcoded in the extension, not user-configurable. TLS protects the WebSocket transport.
- **T4: Popup state manipulation** -- All state is derived from chrome.storage, not React component state. Popup destruction on close does not lose state. Service worker is the source of truth.
- **Mitigations applied:** QR TTL auto-refresh (5 min), single-use pairing tokens, privateKey held only in session storage, relay enforces 2-client room limit.
</threat_model>

- ping: `{ v:1, type:"ping", id:uuid, payload:{} }` -> server responds `{ type:"pong" }`

  <action>
    Create `extension/src/entrypoints/background.ts` using WXT's `defineBackground` wrapper.

    The service worker has four responsibilities:

    **1. WebSocket connection management:**
    - `connect(roomId, relayURL)`: Opens `new WebSocket(relayURL + '?roomId=' + roomId)`.
    - On `ws.onopen`: Send join envelope `{ v:1, type:"join", id:uuid, payload:{} }`. Start 20-second keepalive interval. Set session state `connectionState: "connected"`.
    - On `ws.onclose`: Stop keepalive. Set `connectionState: "disconnected"`. Schedule reconnect with 3-second delay (only if pairing data exists in storage).
    - On `ws.onerror`: Log error. Let `onclose` handle cleanup.
    - `disconnect()`: Clear keepalive, close WebSocket, set `connectionState: "disconnected"`.

    **2. Keepalive ping (20-second interval):**
    - `setInterval` every 20,000ms. Sends `createEnvelope('ping')` as JSON string.
    - Clear interval on disconnect.

    **3. Message routing (ws.onmessage):**
    - Parse JSON, validate `msg.v === 1`.
    - `msg.type === 'joined'`: Log success. No-op.
    - `msg.type === 'pong'`: No-op (keepalive response).
    - `msg.type === 'error'`: Log `msg.payload.code` and `msg.payload.message`.
    - `msg.type === 'pairing_ack'`: Pairing completion handler (see below).
    - Default (opaque forward from iOS): Check if `msg.payload.data` exists. If so, check msg.type:
      - `'code_response'`: Decrypt `base64ToUint8Array(msg.payload.data)` with stored sharedKey via `open()`. Parse JSON for `{ code, requestId }`. Store in session: `{ lastCode: code, codeReceivedAt: Date.now() }`. Set `connectionState: "code_received"`.
      - Any other forwarded type with `payload.data`: Treat as potential pairing acknowledgment from iOS side. This handles the case where iOS joins and sends its first message. If `pendingPairing` exists in session, complete the pairing flow.

    **4. Pairing flow (triggered by popup via chrome.runtime.onMessage):**
    - Listen for `chrome.runtime.onMessage`:
      - `{ type: 'start_pairing', roomId, relayURL }`: Store roomId in session as `pendingPairingRoomId`. Call `connect(roomId, relayURL)`.
      - `{ type: 'complete_pairing', peerPublicKeyBase64 }`: Load `pendingPairing` from session (contains privateKey base64). Call `deriveSharedKey(base64ToUint8Array(privateKey), base64ToUint8Array(peerPublicKeyBase64))`. Save `PairingData { roomId, relayURL, sharedKey: uint8ArrayToBase64(sharedKey), pairedAt: Date.now() }` to chrome.storage.local. Remove `pendingPairing` from session. Set `connectionState: "connected"`.
      - `{ type: 'request_code' }`: Load pairing data. Create code request: `{ id: crypto.randomUUID(), issuer: '', label: '' }` (no account list per D-04 -- iOS presents account selection). Encrypt with `seal(JSON.stringify(request), base64ToUint8Array(sharedKey))`. Send envelope: `createEnvelope('code_request', { data: uint8ArrayToBase64(encrypted) })`.
      - `{ type: 'unpair' }`: Call `clearPairingData()`. Disconnect WebSocket. Set `connectionState: "unpaired"`.
      - `{ type: 'get_state' }`: Return current connection state and pairing status from session/local storage.

    **5. Service worker startup (top-level in defineBackground):**
    - On load, check `loadPairingData()`. If pairing exists, call `connect(pairing.roomId, pairing.relayURL)` to auto-reconnect.

    The relay URL is: `wss://cooperative-respect-production-29f8.up.railway.app`
